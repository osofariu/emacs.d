<div id="outline-container-orgheadline12" class="outline-3">
<h3 id="orgheadline12"><span class="section-number-3">1.2</span> Running tests</h3>
<div class="outline-text-3" id="text-1-2">
</div><ol class="org-ol"><li><a id="orgheadline6"></a>SBT examples<br /><div class="outline-text-4" id="text-1-2-1">
<p>
You can "chain" multiple commands:<br />
</p>
<pre class="example">
sbt clean compile "testOnly TestA TestB"
</pre>

<ul class="org-ul">
<li>This will run the tasks in batch mode, in sequential order<br /></li>
</ul>

<p>
Continuously compile/test as you save your source files:<br />
</p>
<pre class="example">
~ compile , ~ test
~ testOnly *FunSpecSpec // very useful when working on a spec
</pre>
<ul class="org-ul">
<li>this is very cool: you get instant feedback as soon as you save code changes<br /></li>
<li>I use it with IntelliJ, running sbt under IntelliJ terminal window<br /></li>
</ul>
</div></li>

<li><a id="orgheadline7"></a>SBT command-line arguments<br /><div class="outline-text-4" id="text-1-2-2">
<ul class="org-ul">
<li><b><b>clean</b></b> - Deletes all generated files (in the target directory).<br /></li>
<li><b><b>compile</b></b> - Compiles the main sources (in src/main/scala and src/main/java<br />
directories).<br /></li>
<li><b><b>test</b></b> - Compiles and runs all tests.<br /></li>
<li><b><b>console</b></b> - Starts the Scala interpreter with a classpath including the compiled<br />
sources and all dependencies.<br />
<ul class="org-ul">
<li>To return to sbt, type :quit, Ctrl+D (Unix), or Ctrl+Z<br />
(Windows).<br /></li>
</ul></li>
<li><b><b>run &lt;argument&gt;</b></b> - Runs the main class for the project in the same virtual machine as<br />
sbt.<br /></li>
<li><b><b>package</b></b> - Creates a jar file containing the files in src/main/resources and the<br />
classes compiled from src/main/scala and src/main/java.<br /></li>
<li><b><b>help &lt;command&gt;</b></b> - Displays detailed help for the specified command.<br />
<ul class="org-ul">
<li>If no command is<br />
provided, displays brief descriptions of all commands.<br /></li>
</ul></li>
<li><b><b>reload</b></b> - Reloads the build definition (build.sbt, project/*.scala, project/*.sbt<br />
files).<br />
<ul class="org-ul">
<li>Needed if you change the build definition.<br /></li>
</ul></li>
</ul>
</div></li>

<li><a id="orgheadline11"></a>SBT Tasks (from within sbt)<br /><ol class="org-ol"><li><a id="orgheadline8"></a>test<br /><div class="outline-text-5" id="text-1-2-3-1">
<ul class="org-ul">
<li>The test task accepts no command line arguments and runs all tests<br /></li>
</ul>
</div></li>

<li><a id="orgheadline9"></a>testOnly<br /><div class="outline-text-5" id="text-1-2-3-2">
<ul class="org-ul">
<li>this task accepts a whitespace separated list of test names to run.<br /></li>
<li>before version 0.13 this task was "test-only". As of 0.13 they both work.<br /></li>
</ul>

<pre class="example">
testOnly testingscala.tags.FixtureWithSpec testingscala.tags.TagTest
</pre>

<ul class="org-ul">
<li><b>use wildcards</b> to run only certain tests by name:<br /></li>
</ul>
<pre class="example">
testOnly *FixtureWithSpec
</pre>

<ul class="org-ul">
<li><b><b>use tags</b></b> to only run certain tests by tag:<br /></li>
</ul>

<pre class="example">
testOnly *TagTest -- -n FastTest // only run tests tagged as "FastTest"
testOnly *TagTest -- -n SlowTest -n FastTest // run both "SlowTest" and "FastTest" tags
testOnly *TagTest -- -l SlowTest // exclude tests tagged as "SlowTest"
testOnly *TagTest -- -l SlowTest // exclude these tags
</pre>
</div></li>

<li><a id="orgheadline10"></a><b><b>testQuick</b></b> - like testOnly, but also takes into account additional filters:<br /><div class="outline-text-5" id="text-1-2-3-3">
<ul class="org-ul">
<li>The tests that failed in the previous run<br /></li>
<li>The tests that were not run before<br /></li>
<li>The tests that have one or more transitive dependencies, maybe in a different project,<br />
recompiled.<br /></li>
</ul>
</div></li></ol></li></ol>
</div>
</div>

<li><a id="orgheadline18"></a>FunSpec<br /><div class="outline-text-4" id="text-2-1-3">
<ul class="org-ul">
<li>FunSpec allows you to nest tests, so it's clear to see the intention of the tests<br /></li>
</ul>
</div>
<ol class="org-ol"><li><a id="orgheadline16"></a>Example:<br /><div class="outline-text-6" id="text-2-1-3-0-1">
<div class="org-src-container">

<pre class="src src-scala"><span style="color: #0000FF;">import</span> org.scalatest.<span style="color: #D0372D;">FunSpec</span>

<span style="color: #0000FF;">class</span> <span style="color: #6434A3;">SetSpec</span> <span style="color: #0000FF;">extends</span> <span style="color: #6434A3;">FunSpec</span> {

describe(<span style="color: #008000;">"A Set"</span>) {
describe(<span style="color: #008000;">"when empty"</span>) {
it(<span style="color: #008000;">"should have size 0"</span>) {
assert(<span style="color: #D0372D;">Set</span>.empty.size == <span style="color: #D0372D;">0</span>)
}

it(<span style="color: #008000;">"should produce NoSuchElementException when head is invoked"</span>) {
intercept[<span style="color: #D0372D;">NoSuchElementException</span>] {
<span style="color: #D0372D;">Set</span>.empty.head
}
}
}
}
}
</pre>
</div>
</div></li>

<pre class="src src-scala"><span style="color: #0000FF;">import</span> org.scalatest.<span style="color: #0000FF;">_</span>
<span style="color: #0000FF;">import</span> prop.<span style="color: #0000FF;">_</span>
<span style="color: #0000FF;">import</span> scala.collection.immutable.<span style="color: #0000FF;">_</span>

<span style="color: #0000FF;">class</span> <span style="color: #6434A3;">SetSpec</span> <span style="color: #0000FF;">extends</span> <span style="color: #6434A3;">PropSpec</span> <span style="color: #0000FF;">with</span> <span style="color: #6434A3;">TableDrivenPropertyChecks</span> <span style="color: #0000FF;">with</span> <span style="color: #6434A3;">Matchers</span> {

<span style="color: #0000FF;">val</span> <span style="color: #BA36A5;">examples</span> <span style="color: #0000FF;">=</span>
<span style="color: #D0372D;">Table</span>(
<span style="color: #008000;">"set"</span>,
<span style="color: #D0372D;">BitSet</span>.empty,
<span style="color: #D0372D;">HashSet</span>.empty[<span style="color: #D0372D;">Int</span>],
<span style="color: #D0372D;">TreeSet</span>.empty[<span style="color: #D0372D;">Int</span>]
)

property(<span style="color: #008000;">"an empty Set should have size 0"</span>) {
forAll(examples) { set <span style="color: #0000FF;">=&gt;</span>
set.size should be (<span style="color: #D0372D;">0</span>)
}
}

property(<span style="color: #008000;">"invoking head on an empty set should produce NoSuchElementException"</span>) {
forAll(examples) { set <span style="color: #0000FF;">=&gt;</span>
a [<span style="color: #D0372D;">NoSuchElementException</span>] should be thrownBy { set.head }
}
}
}
</pre>

<h3 id="orgheadline26"><span class="section-number-3">2.2</span> ScalaTest (from other preso)</h3>
<div class="outline-text-3" id="text-2-2">
<ul class="org-ul">
<li>interfaces with Junit and TestNG (demonstrate)<br /></li>
<li>two styles of asserts (demonstrate)<br /></li>
<li>several flavors of tests (examples)<br />
<ul class="org-ul">
<li>FunSpec is the most basic<br />
<ul class="org-ul">
<li>BDD - driven specification<br /></li>
<li>uses "describe" and "it"<br /></li>
</ul></li>
</ul></li>
</ul>
</div>
<ol class="org-ol"><li><a id="orgheadline24"></a>Scalactic<br /><div class="outline-text-4" id="text-2-2-1">
<ul class="org-ul">
<li>In 2.2.0, org.scalautils has been renamed to org.scalactic (rhymes with "galactic").<br /></li>
<li>in 3.0 support Scala.JS<br /></li>
</ul>
</div></li>
<li><a id="orgheadline25"></a>3.0<br /><div class="outline-text-4" id="text-2-2-2">
<ul class="org-ul">
<li>now fully support Scala.js<br /></li>
<li>AsyncFunSuite, AsyncFunSpec, AsyncFlatSpec, AsyncFreeSpec, AsyncWordspec, and AsyncFeatureSpec<br />
(the result type of tests is Future[Assertion])<br /></li>
<li>oneElementOf, noElementsOf, atLeastOneElementOf, atMostOneElementOf, allElementsOf, inOrderElementsOf<br /></li>
<li>Added overloaded exists and forEvery methods to TableDrivenPropertyChecks.<br /></li>
</ul>
</div></li></ol>
